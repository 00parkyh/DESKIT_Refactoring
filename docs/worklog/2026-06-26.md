## Manual Seller Login Investigation

### 작업 목적
- seller 데이터를 수동 생성했을 때 로그인 가능한지 확인했다.

### 수정한 파일
- 없음

### 수정 전 코드
- 해당 없음

### 수정 후 코드
- 해당 없음

### 수정 이유
- 코드 변경 없이 로그인 방식, OAuth 처리, seller 조회 조건을 확인했다.

### 확인 결과
- 프론트 로그인은 소셜 OAuth만 제공한다.
- Spring Security는 form login과 HTTP Basic을 비활성화하고 OAuth2/JWT 기반으로 동작한다.
- OAuth 로그인 성공 시 OAuth 이메일이 `seller.login_id`와 일치하면 seller로 로그인된다.
- `seller.status`가 `PENDING`이면 로그인 실패하고, `ACTIVE`면 seller JWT가 발급된다.
- DB에 seller만 수동 생성해도 비밀번호 로그인을 할 수는 없고, 동일 이메일의 OAuth 계정으로 로그인해야 한다.

### 추가로 해야 할 작업
- 로컬 개발용 seller 로그인이 필요하면 dev 전용 인증 엔드포인트 또는 테스트 JWT 발급 기능을 별도 프로파일로 제한해 추가하는 방안을 검토한다.

### 확인 결과
- 방송 시작은 `/api/seller/broadcasts/{broadcastId}/start`에서 현재 로그인한 seller를 조회하고, 방송의 sellerId와 일치해야 가능하다.
- 방송 시작 시 OpenVidu 세션/토큰 생성이 필요하므로 실제 ON_AIR 테스트에는 OpenVidu도 실행되어야 한다.
- 시청자 입장은 `/api/broadcasts/{broadcastId}/join`으로 처리되며 `X-Viewer-Id`가 없으면 UUID 기반 익명 시청자도 허용된다.
- 채팅 송신은 `/pub/chat/message`, 수신은 `/sub/chat/{broadcastId}` 경로를 사용한다.
- 채팅 권한 검사는 memberEmail이 DB member와 매칭되는 경우 제재 여부를 확인하며, memberId가 없으면 기본적으로 채팅 가능하다.

### 추가로 해야 할 작업
- 판매자 1/2, 시청자 1/2를 안정적으로 테스트하려면 dev profile 전용 테스트 로그인/JWT 발급 기능을 추가하는 것이 적절하다.
