# 작업 목적
- 현재 프로젝트의 NCP Object Storage 사용 범위와 S3 API 의존 위치를 확인

# 확인 내용
- S3 관련 핵심 코드는 `S3Config`, `AwsS3Service`, `S3Uploader`에 위치함
- S3 SDK 호출은 `AmazonS3` 기반으로 구현되어 있으며 이미지 업로드, 삭제, URL 생성, VOD 업로드/스트리밍까지 포함함
- S3 관련 사용처는 `UploadController`, `SellerProductController`, `ProductService`, `SetupService`, `VodService`, `BroadcastService`까지 연결되어 있음
- `build.gradle`에는 `aws-java-sdk-s3` 의존성이 선언되어 있음
- `application.properties`에는 `cloud.aws.*` 실제 값이 없고, 코드에서 `@Value`로 직접 주입받고 있음
- 저장소 내 `.env`에는 S3 관련 값이 없어서 실제 `cloud.aws.*` 값은 별도 프로퍼티 파일, IDE 런 설정, 배포 환경 변수 중 하나에서 주입되는 상태로 판단함
- README에는 `cloud.aws.*`를 로컬 프로필에서 교체해야 한다는 안내가 있음

# 판단 정리
- 로컬 파일 저장소로 전환하면 업로드 코드만이 아니라 삭제, 공개 URL 생성, VOD 스트리밍 처리까지 함께 수정해야 함
- 현재 구조에서는 로컬 파일 저장소보다 S3 호환 스토리지로 교체하는 편이 변경 범위가 작음
- MinIO는 S3 호환 스토리지이므로 현재 코드 구조와 잘 맞고, `cloud.aws.*` 변수명은 유지한 채 값만 MinIO 기준으로 교체하는 방식이 가능함
- MinIO 적용 시 필요한 값은 access key, secret key, bucket, endpoint이며 `region`은 코드상 필요하므로 함께 설정해야 함
- 기존 NCP 자격증명이 타인이 발급한 것이라면 장기적으로는 본인 관리 자격증명 또는 자체 운영 스토리지 자격증명으로 교체하는 것이 안전함

# 대화 요약
- 사용자가 NCP Object Storage 대체 방안을 질문함
- 현재 프로젝트는 S3 호환 API에 직접 의존하고 있으므로 단순 로컬 저장 전환은 백엔드 수정 범위가 넓다고 설명함
- MinIO가 S3 호환 오브젝트 스토리지라는 점과, 현재 프로젝트에 비교적 자연스럽게 적용 가능한 대안이라는 점을 정리함
- S3 관련 코드 위치, S3 API 호출 위치, 설정 주입 구조를 확인해 설명함
- 사용자가 보유한 기존 NCP 환경 변수 예시를 기준으로 MinIO에서도 유사한 매핑이 가능하다고 안내함
- MinIO 설치 시 `MINIO_ROOT_USER`, `MINIO_ROOT_PASSWORD`로 시작하고, 이후 bucket 및 access key/secret key를 현재 `CLOUD_AWS_*` 환경 변수에 매핑해 사용할 수 있다고 정리함
